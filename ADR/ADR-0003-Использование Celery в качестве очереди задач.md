# Использование Celery в качестве очереди задач

Дата: 2023-10-01

## Статус

Принято

## Контекст

Наша задача реалезовать serverless приложение, которое бы могло запускать чужие функции на нашем железе
Возникла необходимость обрабатывать поступающие запросы так что бы выполнение этих запросов выполнялось асинхронно, не прерывая основной цикл программы

### Варианты

1. Celery. Асинхронная очередь задач с открытым исходным кодом или очередь заданий, основанная на распределенной передаче сообщений. Хотя он поддерживает планирование, его основное внимание уделяется операциям в реальном времени.
2. Apache Kafka. Распределённый программный брокер сообщений с открытым исходным кодом, который разрабатывается в рамках фонда Apache на языках Java и Scala.
Цель проекта — создание горизонтально масштабируемой платформы для обработки потоковых данных в реальном времени с высокой пропускной способностью и низкой задержкой.
3. RabbitMQ. Программный брокер сообщений, основанный на стандарте AMQP. Это тиражируемое связующее программное обеспечение, ориентированное на обработку сообщений.
Изначально разрабатывался компанией SpringSource, после серии поглощений и разделений вошедшей в состав Pivotal. Выпускается под Mozilla Public License.

## Решение

Выбранный вариант: "[Celery]", потому что celery под капотом сделан на Python, что упращает создание новых задач, из за наличия хорошего разработчика. Сложность в не знании/не умении пользоваться RabbitMQ и Kafka, а так же необходимость писать под них свои отдельные обработчики, что пагубно может повлеять на время реализации.

## Последствия

Проблем во время эксплоатации не обнаружено.
